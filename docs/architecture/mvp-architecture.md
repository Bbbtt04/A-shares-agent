# 股票情报 Agent MVP 架构

本文档定义第一阶段 MVP 架构。MVP 的目标不是一次性实现完整多 Agent 系统，而是建立一条可用、可追溯、可复盘的最小股票情报闭环。

## 1. MVP 定位

MVP 定位为 A 股情报辅助系统，默认只提供分析、提醒、模拟信号和复盘，不接入真实交易。

MVP 必须回答三个问题：

- 盘前今天应该重点看什么？
- 盘中为什么某个股票或板块正在异动？
- 盘后今天的判断对不对，哪里错了？

## 2. MVP 范围

第一阶段只做四条核心链路：

```text
数据接入
  -> 盘前情报生成
  -> 盘中异动解释
  -> 盘后复盘
```

暂不做：

- 真实交易接入。
- 完全自主多 Agent 协商。
- 复杂实时盘口策略。
- 大规模研报理解。
- 个性化策略自动优化。
- 自动修改策略参数。

## 3. MVP 模块

### 3.1 数据接入模块

最小数据范围：

- A 股日线与分钟级行情。
- 板块行情。
- 涨跌停与涨速数据。
- 上市公司公告。
- 财经新闻。
- 用户关注股票池。

基础治理能力：

- 股票代码标准化。
- 新闻与公告去重。
- 数据源记录。
- 发布时间记录。
- 简单数据质量检查。

MVP 阶段不追求全量数据源，优先保证少量可靠来源稳定可用。

### 3.2 知识与状态模块

MVP 需要两类存储。

结构化状态：

- 今日关注股票池。
- 今日主题与板块列表。
- 盘前策略假设。
- 盘中提醒记录。
- 个股状态。
- 盘后复盘结果。

轻量 RAG：

- 公告摘要。
- 新闻事件摘要。
- 公司基础画像。
- 题材与概念说明。
- 历史复盘结论。

MVP 可优先使用 PostgreSQL 加 pgvector，或者 PostgreSQL 结构化表加全文搜索。只有当检索规模和效果成为瓶颈时再引入独立向量库或搜索集群。

### 3.3 LLM 网关

MVP 需要一个轻量 LLM Gateway，不必一开始做成复杂平台。

必备能力：

- 统一模型调用入口。
- Prompt 模板集中管理。
- JSON schema 输出校验。
- 调用日志。
- Token 和成本记录。
- 超时与重试。
- 简单模型路由。

MVP 推荐至少区分三类任务：

- 摘要类：新闻与公告摘要。
- 抽取类：公告字段、事件主体、影响标的。
- 推理类：盘前策略、盘中解释、盘后复盘。

### 3.4 Tool Registry

MVP 工具注册表至少支持：

- `get_market_snapshot`：查询行情快照。
- `get_stock_history`：查询历史行情。
- `get_sector_snapshot`：查询板块行情。
- `search_news`：查询新闻。
- `search_announcements`：查询公告。
- `query_knowledge`：查询知识库。
- `write_agent_event`：写入 Agent 事件。
- `send_alert`：发送提醒，可先只写入本地提醒队列。

每个工具必须声明输入、输出、权限和超时。盘中链路工具尤其需要设置超时，避免实时任务被慢数据源拖住。

### 3.5 沙盒与权限

MVP 可以先实现简单权限模型。

建议角色：

- `reader`：只读行情、新闻、公告、知识库。
- `analyst`：可调用 LLM 和计算工具。
- `state_writer`：可写入关注池、提醒和复盘状态。
- `notifier`：可发送提醒。

Agent 权限建议：

| Agent | 权限 |
| --- | --- |
| 盘前策略 Agent | `reader`, `analyst`, `state_writer` |
| 盘中策略 Agent | `reader`, `analyst`, `state_writer`, `notifier` |
| 公告解读 Agent | `reader`, `analyst`, `state_writer` |
| 新闻情报 Agent | `reader`, `analyst`, `state_writer` |
| 复盘 Agent | `reader`, `analyst`, `state_writer` |

MVP 阶段不开放任何真实交易权限。

### 3.6 监控与日志

MVP 必备监控：

- 每个任务是否成功。
- 每个工具调用耗时与错误。
- LLM 调用耗时、失败、成本。
- 盘前报告是否按时生成。
- 盘中提醒数量。
- 复盘是否完成。

MVP 必备日志：

- Agent 输入摘要。
- Agent 输出。
- 使用的证据。
- 工具调用记录。
- 失败原因。

## 4. MVP Agent

### 4.1 盘前策略 Agent

职责：

- 汇总隔夜市场、新闻、公告和昨日市场结构。
- 生成今日市场环境判断。
- 生成主线题材排序。
- 生成关注股票池与回避股票池。
- 生成盘中可复用的策略假设。

输出结构：

```text
市场环境
主线题材
关注股票池
回避风险
盘中观察条件
证据列表
```

### 4.2 新闻情报 Agent

职责：

- 聚合新闻。
- 去重。
- 识别事件。
- 标注关联股票、行业、题材。
- 给出初步影响方向。

输出结构：

```text
事件标题
事件摘要
关联标的
关联题材
影响方向
可信度
来源
```

### 4.3 公告解读 Agent

职责：

- 解析上市公司公告。
- 提取公告类型、关键数值、影响对象。
- 判断是否需要进入盘前关注或盘中提醒。

输出结构：

```text
公告类型
关键事实
关键数值
影响判断
风险点
证据
```

### 4.4 盘中策略 Agent

职责：

- 接收行情异动事件。
- 查询新闻、公告、板块和盘前假设。
- 判断异动原因。
- 输出提醒和状态更新。

输出结构：

```text
异动标的
异动类型
异动解释
是否匹配盘前假设
提醒等级
后续观察条件
风险
证据
```

### 4.5 复盘 Agent

职责：

- 对盘前观点、盘中提醒和市场结果进行复盘。
- 判断哪些假设被验证，哪些被证伪。
- 归因错误来源。
- 沉淀复盘结论。

输出结构：

```text
盘前判断回顾
盘中提醒回顾
命中案例
失败案例
错误归因
明日改进建议
```

## 5. MVP 工作流

### 5.1 盘前工作流

```text
定时触发
  -> 拉取新闻、公告、隔夜市场和昨日行情
  -> 新闻情报 Agent 生成事件列表
  -> 公告解读 Agent 生成公告事件
  -> 盘前策略 Agent 生成盘前报告和策略假设
  -> 写入今日状态
  -> 输出盘前日报
```

### 5.2 盘中工作流

```text
行情异动触发
  -> 生成异动事件
  -> 查询盘前策略假设
  -> 查询相关新闻、公告、板块联动
  -> 盘中策略 Agent 生成解释
  -> 写入提醒记录
  -> 必要时发送提醒
```

### 5.3 盘后复盘工作流

```text
收盘后触发
  -> 汇总今日盘前报告、盘中提醒、行情结果
  -> 复盘 Agent 评估判断质量
  -> 生成复盘报告
  -> 写入复盘知识库
```

## 6. MVP 数据模型建议

核心实体：

- `Stock`：股票基础信息。
- `MarketSnapshot`：行情快照。
- `NewsEvent`：新闻事件。
- `AnnouncementEvent`：公告事件。
- `WatchlistItem`：关注池条目。
- `PremarketReport`：盘前报告。
- `PremarketHypothesis`：盘前策略假设。
- `IntradayAlert`：盘中提醒。
- `AgentTrace`：Agent 调用轨迹。
- `ReviewReport`：复盘报告。

关键字段建议：

- 所有事件必须有 `source`, `published_at`, `ingested_at`, `confidence`。
- 所有 Agent 输出必须有 `evidence_ids`。
- 所有提醒必须有 `severity`, `reason`, `risk`, `created_at`。
- 所有策略假设必须有 `invalidate_condition`，用于盘中证伪。

## 7. MVP 验收标准

功能验收：

- 每个交易日前能生成一份盘前报告。
- 盘前报告包含市场环境、主线题材、关注池、风险和证据。
- 盘中能对至少一种异动事件生成解释。
- 盘中提醒能关联新闻、公告、板块或盘前假设。
- 每日能生成复盘报告。

工程验收：

- Agent 输出为结构化数据。
- 关键工具调用有日志。
- 关键结论有证据 ID。
- LLM 调用有成本与失败记录。
- 盘中链路有超时控制。
- 不存在真实交易权限。

质量验收：

- 公告关键字段抽取可人工抽检。
- 新闻事件去重可人工抽检。
- 盘中提醒必须能解释触发原因。
- 复盘能指出错误来源，而不只是描述涨跌结果。

## 8. MVP 推荐技术栈

建议沿用轻量、可维护的技术栈：

- 后端：Python FastAPI。
- 任务调度：APScheduler 或 Celery Beat。
- 异步任务：Celery、Dramatiq 或 RQ。
- 数据库：PostgreSQL。
- 缓存与盘中状态：Redis。
- RAG：PostgreSQL full-text + pgvector 起步。
- 前端：React 或 Vite。
- 监控：结构化日志 + 基础 metrics，后续接 Prometheus 和 Grafana。

## 9. 第一阶段优先级

推荐实施顺序：

1. 数据接入与标准化。
2. 轻量 LLM 网关。
3. Tool Registry 与权限。
4. 新闻情报 Agent。
5. 公告解读 Agent。
6. 盘前策略 Agent。
7. 盘中异动解释。
8. 盘后复盘。
9. 监控看板与人工反馈。

MVP 成功的标志不是 Agent 看起来很自主，而是每天稳定地产生可追溯、可验证、可复盘的情报结果。
